# Veylbase Phase 005/006 Plan Review

Consolidated from four adversarial lenses (Completeness, Technical Correctness, GSD Conformance, Ship-Gate Rigor). Disputed claims were re-verified against the installed code and the `@zama-fhe/sdk` 3.1.0 dist before merging. File anchors are absolute-equivalent repo paths.

## Verdict

The 005/006 plan set is disciplined and largely correct: artifacts are GSD-conformant, the COOP/COEP "do-not-add" decision is verified-correct (`web()` is called argument-less at `confidential-actions.ts:115`, single-thread, no SharedArrayBuffer), the WETH 18/6 decimals case is rightly singled out as the mandatory gate, and the SDK 3.1->3.2 bump is feasible (installed viem 2.53.1 satisfies 3.2.0's `viem ^2.47.4`). It is NOT yet safe to hand off as-is: the "highest-value" resume-unshield fix is wired against a false premise and is a silent no-op as written, and two schedule-structural gaps (portal registration back-loaded, no dated critical path) are the most likely ways a working build still misses the deadline. Fix the three blockers below and reconcile the cross-cutting CSP/RPC/relayer-host triangle, and the plan becomes trustworthy.

## Blockers (must fix before the plan is trustworthy)

### B1. Resume-unshield is wired against a false SDK premise — no-op as planned
- **Issue:** The plan (005 PLAN step 9) and brief assert "the SDK auto-persists the unwrap, so `loadPendingUnshield` returns a pending hash on reconnect." This is FALSE in the installed dist. Verified in `node_modules/@zama-fhe/sdk/dist/esm/index.js`: `unshield(e)` runs `this.unwrap(e)` then the private finalize `#s(c.txHash, s, o)` and NEVER calls `savePendingUnshield`. `savePendingUnshield` appears exactly once in the whole dist — as a bare re-export alias (`on as savePendingUnshield`) with zero internal call sites. Therefore `loadPendingUnshield(storage, wrapperAddress)` always returns `null`, and the planned "Resume pending unshield" banner can never render. UAT E06 would look implemented and silently never fire. (The ship-gate lens called this premise "real because the functions exist in 3.1.0" — the functions exist, but that misses the orchestration gap; the technical-correctness lens is right, the ship-gate lens is wrong on this point.)
- **Exact fix:** Make the APP persist the hash. In `src/app/confidential-actions.ts` `executeUnshield`, inside the existing `onUnwrapSubmitted` callback (line ~253) call `savePendingUnshield(indexedDBStorage, wrapperAddress, txHash)` (import from `@zama-fhe/sdk`), and on successful finalize call `clearPendingUnshield(indexedDBStorage, wrapperAddress)`. Add a bridge fn `executeResumeUnshield` that calls `wrapper.resumeUnshield(hash)` — verified to work from a hash alone (`resumeUnshield(e,t){return this.#s(e, crypto.randomUUID(), t)}`, reloads the receipt and finalizes). Use the SAME `indexedDBStorage` instance and SAME `wrapperAddress` the SDK uses so keys match. The VERIFICATION synthetic-pending-entry test must write via `savePendingUnshield` (not a hand-rolled key) so it exercises the real path.
- **File:** `src/app/confidential-actions.ts`; rewrite `.planning/phases/005-production-hardening-docs-and-demo/PLAN.md` step 9; correct the premise line in the research brief / CONTEXT risk.

State a hard checkpoint: file by 2026-07-06 23:59 (24h pre-deadline); if the form is not in hand by 2026-07-04, escalate to developer@zama.org / @zama_fhe. Record the form fields into UAT.md so nothing is discovered late.

### B3. No dated critical path / freeze point — single-owner serialization with no slack
- **Issue:** Both phases are tiered (MUST/SHOULD/NICE) but neither assigns DATES. The whole 006 critical path (provision RPC -> deploy -> smoke -> 19-case UAT with ~5 on-chain txs each + live-relayer waits -> defect fix -> re-UAT -> real-voice demo -> register -> submit) is serialized on one person who also owns 005's SHOULD code. A single FAIL (decimals or resume defect) reopens 005 and re-runs the gate. There is no stated "point of no return."
- **Exact fix:** Add a dated schedule, e.g. Day1-2 = 005 MUST + deploy + portal registration in parallel; Day3 = 005 SHOULD (resume-unshield, auto-rehide, error-UX); Day4 = full UAT; Day5 = defect fixes + re-UAT; Day6 = demo; Day7 = buffer, submit by Day6 EOD. State the cut order if behind: drop NICE (CSP, decimals assertion) first, then SHOULD #11 retry-button, then SHOULD #9 resume-unshield (downgraded to documented known-limitation) — NEVER cut the decimals UAT, LICENSE, live-URL, or video deliverables. Name the freeze: "no new code after Day5 EOD; remaining UAT FAILs become documented known limitations."

## De-duplicated cross-cutting themes

Do this once; it discharges M2/M4/M6/M7.

- **react-sdk unused was flagged twice (technical + completeness) — one fix** (N1).

- **SDK-version sequencing ties B1, M5, and N1 together.** Bump to 3.2.0 first, read its changelog for the persistence + callback APIs, then build the app-side `savePendingUnshield`/`resumeUnshield` wiring against the shipping version, and either drop or annotate react-sdk in the same edit.
